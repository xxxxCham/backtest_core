# Analyse : Sélection répétée du token 0GUSDC

## Résumé du problème
Le système Strategy Builder semble sélectionner **0GUSDC** de manière excessive, malgré les consignes de diversification implémentées dans le code.

## Mécanismes de diversification existants

### 1. Shuffle des symboles (Réduction du biais de position)
**Fichier** : `agents/builder_objectives.py`

**Lignes 564-565** (dans `recommend_market_context`) :
```python
shuffled_symbols = symbols.copy()
random.shuffle(shuffled_symbols)
```

**Lignes 282-283** (dans `generate_llm_objective`) :
```python
shuffled_symbols = symbols_list.copy()
random.shuffle(shuffled_symbols)
```

✅ **Verdict** : Le shuffle est bien implémenté pour éviter que BTC/premier symbole soit toujours favorisé.

## Hypothèses sur la cause

### ❌ Hypothèse 1 : `auto_market_pick` désactivé
Si `auto_market_pick = False`, le système n'utilise **PAS** `recommend_market_context()` et reste sur le symbole/timeframe par défaut de la sidebar.

**Vérification requise** :
- Ligne 1195 dans `builder_view.py` affiche "Auto marché: ON" ou "OFF"
- Regardez dans l'UI si cette option est activée

---

### ❌ Hypothèse 2 : Univers de symboles limité
Si `all_symbols` ne contient que très peu de symboles (ex: 3-4 tokens), la rotation devient mécaniquement limitée.

**Vérification requise** :
- Ligne 978 : `all_symbols = user_symbols if user_symbols else (available_tokens[:20]...)`
- Combien de symboles sont dans `available_tokens` ?
- Y a-t-il une sélection manuelle dans la sidebar qui restreint l'univers ?

---

### ⚠️ Hypothèse 3 : Biais intrinsèque du LLM
Malgré le shuffle, le modèle peut avoir développé un **biais positionnel** ou **sémantique** :
- **0GUSDC** commence par "0" → position lexicographique particulière
- Le shuffle est `random.shuffle()` mais toujours appliqué de la même façon
- Le modèle peut "apprendre" que 0GUSDC est acceptable et s'y fixer

**Test de diagnostic** :
1. Vérifier les logs de `recommend_market_context` pour voir si :
   - Le LLM choisit **vraiment** 0GUSDC à chaque fois, ou
   - Si l'override anti-répétition est déclenché mais sans effet
2. Regarder le contenu exact du prompt envoyé au LLM (avec le shuffle)

---

### ⚠️ Hypothèse 4 : L'historique ne se propage pas correctement
Si `history` est vidé/réinitialisé entre les runs, `_recent_markets` sera toujours vide.

**Vérification requise** :
- Ligne 1391 : `st.session_state["builder_autonomous_history"] = history`
- Vérifier si `history` persiste bien entre les itérations de la boucle `while`
- Ajouter un log temporaire pour afficher `len(_recent_markets)` avant chaque génération

---
